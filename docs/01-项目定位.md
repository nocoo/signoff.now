# 01 — 项目定位

> 状态：已确认  
> 范围：产品定位、核心概念、实体层级、采集边界、质量标准、一期/非一期  
> 后续文档：实体与 Schema、活跃度计分细则、CLI/Scripts 管线、Web/D1 架构等

---

## 1. 一句话

**signoff.now** 是一个围绕**开发者**与 **Git 仓库**的数据可视化与分析平台：在本机用 CLI / Scripts / Skills 从 Azure DevOps 拉取可验证的数据，写入 Cloudflare D1；Web 前端只读展示，供**管理者**查看团队活跃度、热力图与多选对比。

---

## 2. 产品形态

| 组成部分 | 职责 |
|:---------|:-----|
| **Web** | Vite + React 前端；部署于 Cloudflare Worker；Cloudflare Access 认证；**只读** D1 结果；设置与实体管理 UI |
| **CLI** | 语义化、可单测的数据获取与变换（标杆：现有 `gitinfo` / `pulse` 的质量标准） |
| **Scripts** | 串流程、落盘编排、schema 校验、ingest 到 D1 等「流程锁定」逻辑 |
| **Skills（项目级）** | Agent 编排层：调用 CLI/Scripts，**禁止**手写或改写 JSON 结构；保证多次运行 schema 不漂移 |

### 2.1 运行拓扑

```
本机（有 az 权限）                         云端（Cloudflare）
┌─────────────────────────────┐           ┌──────────────────────────┐
│  Skill → CLI → Scripts      │  ingest   │  Worker (SPA + API)      │
│  落盘 JSON + 校验            │ ───────►  │  D1（业务数据）          │
│  验证 az 已登录              │           │  Access 保护              │
└─────────────────────────────┘           │  主要为「查看与配置」      │
                                          └──────────────────────────┘
```

- **采集只在本机**：ADO 依赖本机已登录的 `az`；启动管线前必须先校验登录状态。
- **云端以查看为主**：Repo 绑定、Developer/Team/Tag、权重与时区等配置可在 Web 设置页管理并落 D1；**不在云端跑 az 拉数**。
- **不做多租户 SaaS**：放弃「多用户数据隔离」产品形态。开源部署者各自部署一份实例；认证用 Access 挡住未授权访问即可，**不设计** Workspace/Tenant 多套账本。

### 2.2 与历史方案的关系

与旧版「AI Code Review / Story / Wisdom」叙事**无关**，不继承其产品目标与文档主线。现有 monorepo 中的 Electron 桌面、历史 archive 文档仅作考古，不约束本定位。

## 3. 用户与使用场景

| 角色 | 场景 |
|:-----|:-----|
| **管理者（主用户）** | 维护开发者花名册与仓库绑定；按时间范围查看活跃度热力图与明细；多选开发者做简单维度对比；调整积分权重与时区 |
| **部署者** | 自托管 Worker + D1 + Access；本机配置 `az` 后跑采集 skill/scripts |

一期不强调「开发者只看自己」的个人门户。

## 4. 核心概念

### 4.1 Developer（开发者）

| 字段概念 | 说明 |
|:---------|:-----|
| **name** | 显示名（Display name），与 alias **分开存储** |
| **alias** | 短标识（非邮箱本体）；与设置中的**邮箱后缀**拼接成完整邮箱，用于匹配 git / ADO 身份（如 `uniqueName`） |
| **tags** | 多对多：挂到独立 Tag 实体，用于筛选与对比勾选 |
| **teams** | 多对多：一人可属多个 Team |

身份匹配原则：`alias + "@" + emailSuffix`（及后续可能扩展的显式邮箱列表，细节见实体文档）优先对齐 ADO `createdBy.uniqueName` / reviewer `uniqueName` 等稳定字段。

### 4.2 Repo（仓库）

- 绑定粒度：**单个 Git repository**（不是整个 ADO Project）。
- 一期 **provider = Azure DevOps**；GitHub **二期**。
- 绑定信息存 **D1**（org / project / repo 名、URL、外部 id 等），供 Web 展示与本机采集清单使用。
- 采集始终在本机对「已绑定列表」执行。

参考本机 work 目录现实情况：多 org（如 多个 Azure DevOps organization）必须支持，实体模型不能假设单一 organization。

### 4.3 Team（团队）

- 原「Group」统一命名为 **Team**。
- 开发者与 Team：**多对多**。
- 用于组织视图与筛选；不与 Tag 混用。

### 4.4 Tag（标签）

- **独立实体**，不是 Developer 上的自由字符串数组。
- 每个 Tag 有名称与**颜色**，可赋予多个 Developer。
- 典型用途：对比前快速筛选（如「后端」「实习」「Oncall」）。

### 4.5 Activity（活动）

- 原 ContributionEvent 统一为 **Activity**。
- 表示「某开发者在某时刻、某仓库（或全局 WI）上发生的一条可计分事件」。
- 一期来源仅限 **ADO 的 PR 与 Work Item**（含 PR 上的个人 vote 作为 PR 社交子类）。
- 字段必须来自 **CLI/Scripts 能稳定拿到** 的数据；拿不到或不稳则 **宁可丢弃该 Activity / 不计 Score**，禁止猜测补全。

### 4.6 Score（得分）

- 原 DailyActivityScore 简化为 **Score**。
- 语义：在给定时区下，**某个开发者、某一自然日** 的活跃度积分（可由当日 Activity 加权汇总；是否物化表由后续 schema 文档决定）。
- 权重在 **设置页可调**；下文给出**默认权重**（可被配置覆盖）。

### 4.7 时间

- 自然日边界由 **设置中的时区** 决定。
- **默认时区：`Asia/Shanghai`**。
- 热力图与列表的时间范围可调（仿 GitHub 贡献热力图交互，具体 UI 后续文档）。

### 4.8 设置（Settings）

Web 提供设置页，至少包括：

| 项 | 说明 |
|:---|:-----|
| 邮箱后缀 | 与 Developer.alias 拼邮箱 |
| 时区 | 默认上海 |
| 活跃度权重 | 覆盖默认权重表 |
| （后续） | ADO 默认同步窗口等 |

## 5. 实体层级（概念模型）

中立命名，便于日后兼容 GitHub；**一期只实现 ADO 采集**。

```
Instance（单次部署，无多租户账本）
├── Settings
├── Developer
│     ├── tags[] → Tag
│     └── teams[] → Team
├── Team
├── Tag
├── Repo                    # 单仓库绑定，D1 存储
├── Activity                # 中立事件流（可扩展 type）
└── Score                   # 按开发者 × 自然日
```

### 5.1 CRUD 与关联原则

1. **先实体、后关联**：Developer / Team / Tag / Repo 均具备完整 CRUD。
2. **再关联修改**：如 Developer↔Team、Developer↔Tag、Repo 启用/停用采集。
3. **Activity / Score** 主要由本机管线写入；Web 以读为主（是否允许手工纠偏后续再定，一期默认只读）。
4. **数据结构与 schema 中立**：存储层不直出 ADO REST 原始字段名；映射层把 ADO（及未来 GitHub）归一到中立 Activity / Repo 等模型。

### 5.2 预留扩展（空实现即可）

以下模块在信息架构与 Activity 类型体系中**预留扩展点**，一期可空实现：

| 模块 | 意图 |
|:-----|:-----|
| **Livesite** | 线上事故处理相关贡献/负担 |
| **Teams / IM** | 即时通讯协作信号（如 Teams 交互） |

活跃度设计为 **可扩展**：新来源 = 新的 `Activity.type`（或 provider 子类型）+ 可选权重键，不推翻 Score 汇总框架。

## 6. 一期活跃度（ADO only）

### 6.1 计入范围

| 大类 | 事件（概念 type） | 说明 |
|:-----|:------------------|:-----|
| **PR** | `pr.merged` | 当日合并完成 |
| **PR** | `pr.closed` | 当日关闭且未合并（废弃/拒绝等） |
| **PR** | `pr.draft` | 当日仍为 draft 且有可稳定观测的活动（见稳定性原则） |
| **PR** | `pr.active` | 当日有更新但未落入更明确终态事件时的「有推进」 |
| **PR** | `pr.vote` | **个人** reviewer 投票（按票记到投票者当日；忽略 group/container reviewer） |
| **Work Item** | `wi.created` / `wi.updated` / `wi.closed` | **全部 work item 类型**均计；以稳定字段的创建/变更/关闭时间为准 |

一期**不包含**：commits 计数、GitHub 任何来源、IM、Livesite（仅预留）。

### 6.2 稳定性原则（高于完整率）

- 只使用 **CLI/Scripts 易获取且稳定** 的字段（例如 `creationDate`、`closedDate`、`status`、`isDraft`、`reviewers[].vote` + `uniqueName`、WI 的 Created/Changed/Closed 等）。
- 若某信号需要昂贵或不稳 API（例如全量 threads 才能推断的「活跃」），**一期可不做或降级丢弃**。
- **Activity 与 Score 都可以少，不能错、不能飘。**

### 6.3 默认权重

设置页可覆盖；下列为出厂默认（整数，便于展示与心算）：

| Activity type | 默认权重 | 备注 |
|:--------------|--------:|:-----|
| `pr.merged` | 10 | 交付完成 |
| `pr.closed` | 2 | 关闭未合并，计参与但不鼓励无效 PR |
| `pr.draft` | 1 | 草稿推进 |
| `pr.active` | 2 | 当日有推进（与终态同时发生时：只计更高优先级终态，避免重复暴击；细则见后续计分文档） |
| `pr.vote` | 3 | 个人有效投票（vote ≠ 0）；group 票忽略 |
| `wi.created` | 3 | |
| `wi.updated` | 1 | 同日多条 updated 是否去重：后续计分文档定；默认倾向「按日每 WI 最多计一次 updated」 |
| `wi.closed` | 5 | |

**同日同一 PR 的优先级（默认）**：`merged` > `closed` > `active` / `draft`（终态优先；vote 单独按投票者计，与作者侧事件独立）。

### 6.4 展示与对比（一期）

| 能力 | 一期要求 |
|:-----|:---------|
| 单人详情 | 热力图 + 明细列表 + 基础可视化（Recharts） |
| **多选对比** | **必须支持**；一期做**简单比较**（如总分、分项汇总、并排热力/柱状）即可 |
| Tag / Team 筛选 | 用于缩小多选集合 |

## 7. 数据管线：CLI / Scripts / Skills

### 7.1 分工

| 层 | 做什么 | 不做什么 |
|:---|:-------|:---------|
| **CLI** | 单一语义命令：如检查 `az` 登录、拉某 repo 时间窗内 PR 列表、归一化映射、输出 JSON 到 stdout 或约定路径 | 不写「半套 schema」；不静默吞掉校验失败 |
| **Scripts** | 串多步 CLI、目录约定、**schema 校验**、失败即非零退出、ingest D1 | 不塞难测的大段业务分支（业务应下沉 CLI 并单测） |
| **Skills** | 编排上述命令的人机/Agent 流程 | **不**直接拼 JSON 字段；不绕过校验 |

`gitinfo`、`pulse` 作为**质量标杆**（DI、JSON-first、高覆盖率、清晰 exit code）。二者语义偏平台中立（本地 git / 远程协作），后续新 CLI 对齐同一工程标准；ADO 主路径可新建语义 CLI，不必扭曲现有命令的 GitHub 假设。

### 7.2 落盘与校验（硬约束）

1. 落盘格式：**仅 JSON**。  
2. **严格目录与文件名**（具体树在后续「采集管线」文档规定；原则示例）：

   ```text
   .data/                          # 不提交（gitignore）
     raw/
       ado/{org}/{project}/{repo}/
         prs/{yyyy}/{yyyy-mm-dd}.json
         votes/...
         workitems/...
     normalized/
       activities/{yyyy-mm-dd}.json
     meta/
       runs/{run-id}.json
   ```

3. 每个产物带 **`schemaVersion`**；仓库内维护 JSON Schema（或等价严格校验器）。  
4. 落盘后 **必须** 跑验证脚本；验证失败 → **禁止** ingest。  
5. `.data/`（或最终选定的数据根）**不得提交**到 git。

### 7.3 ADO 认证

- 使用本机已登录的 **`az`**（及 azure-devops 扩展）。  
- 管线入口 **先** `az` 登录/账号校验，失败则清晰报错退出。  
- 不在一期做「把 PAT 存进云端由 Worker 拉数」。

## 8. Web 与数据存储

| 项 | 决策 |
|:---|:-----|
| 前端 | Vite + React + TypeScript；可视化 **Recharts** |
| 部署 | Cloudflare Worker 托管 SPA + API |
| 认证 | Cloudflare Access |
| 数据库 | Cloudflare D1 |
| 云端角色 | 读 Score/Activity/实体；管理 Developer/Team/Tag/Repo/Settings |
| 本机角色 | 采集、校验、写入 D1（通过受控 scripts） |
| 架构风格 | **严格 MVVM**（Model / ViewModel / View 分层；View 薄） |

多用户「每用户一套数据」**不做**；单实例内管理者维护一套花名册与仓库即可。

## 9. 工程质量

| 项 | 标准 |
|:---|:-----|
| 方法论 | 测试驱动开发（TDD） |
| Lint | **0 error / 0 warning**（Biome `--error-on-warnings`） |
| 覆盖率 | **≥ 95%**（branches / lines / functions） |
| 覆盖率范围 | **CLI、Scripts、shared、Web 的 Model 与 ViewModel 强制**；**View 组件排除**（约定目录如 `**/views/**` 或 `**/*.view.tsx`，在工程配置中钉死） |
| Schema | CLI 输出与落盘 JSON 有自动化校验，防止 Skill/模型漂移 |

## 10. 一期范围与非目标

### 10.1 一期做

- 实体 CRUD：Developer、Team、Tag、Repo、Settings  
- 本机 ADO 采集：PR + Work Item（+ 个人 vote）→ Activity → Score  
- 落盘 JSON + schema 校验 + ingest D1  
- Web：热力图、明细、简单多选对比、设置（时区/后缀/权重）  
- 单实例部署：Worker + D1 + Access  
- Livesite / Teams：**预留扩展点，空实现**

### 10.2 一期明确不做

- 多租户 / 多用户数据隔离 SaaS  
- GitHub 采集与计分（二期）  
- 云端持 PAT 直接拉 ADO  
- Commit 级活跃度  
- 完整 Livesite / IM 业务  
- AI Code Review / Story / Wisdom 旧叙事  
- 移动端  
- 实时 Webhook 驱动计分（一期以本机批量/时间窗拉取为准）  
- 为追求完整率而使用不稳定字段「估」Activity
