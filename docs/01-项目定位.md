# 01 — 项目定位

> 状态：已确认（含审查修订）  
> 范围：产品定位、核心概念、实体层级、采集边界、质量标准、一期/非一期  
> 后续文档：实体与 Schema、活跃度计分细则、CLI/Scripts 管线、Web/D1 架构等

---

## 1. 一句话

**signoff.now** 是一个围绕**开发者**与 **Git 仓库**的数据可视化与分析平台：在本机用 CLI / Scripts / Skills 从 Azure DevOps 拉取可验证的数据，写入 Cloudflare D1；Web 前端供**管理者**配置实体与设置，并**只读**展示 Activity / Score（热力图、明细、多选对比）。

---

## 2. 产品形态

| 组成部分 | 职责 |
|:---------|:-----|
| **Web** | Vite + React；部署于 Cloudflare Worker；Cloudflare Access 认证；**可写**实体与设置；**只读** Activity / Score |
| **CLI** | 语义化、可单测的数据获取与变换（质量标杆：`gitinfo` / `pulse`） |
| **Scripts** | 串流程、落盘编排、schema 校验、ingest 到 D1 |
| **Skills（项目级）** | Agent 编排：只调用 CLI/Scripts，**禁止**手写或改写 JSON 结构 |

### 2.1 运行拓扑

```
本机（有 az 权限）                         云端（Cloudflare）
┌─────────────────────────────┐           ┌──────────────────────────────┐
│  Skill → CLI → Scripts      │  ingest   │  Worker (SPA + API)          │
│  落盘 JSON + 校验            │ ───────►  │  D1                          │
│  验证 az 已登录              │           │  Access 保护                  │
└─────────────────────────────┘           │  写：实体/设置  读：Activity/Score │
                                          └──────────────────────────────┘
```

- **采集只在本机**：ADO 依赖本机已登录的 `az`；管线入口必须先校验登录状态。
- **云端职责拆分**（避免「只读」歧义）：
  - **可写（管理）**：Developer / Team / Tag / Repo 绑定 / Settings（时区、权重、邮箱后缀等）
  - **只读（分析数据）**：**Activity** 与 **Score** 仅由本机管线写入；Web **不得**手工编造或改写计分事件
- **不做多租户 SaaS**：单实例部署；Access 仅做访问控制，不设计 Workspace/Tenant 多套账本。

### 2.2 与历史方案的关系

与旧版「AI Code Review / Story / Wisdom」叙事**无关**。Electron 桌面、本地 SQLite（better-sqlite3 / 旧 `local-db`）均已退役；持久化以 **Cloudflare D1** 为准。

## 3. 用户与使用场景

| 角色 | 场景 |
|:-----|:-----|
| **管理者（主用户）** | 维护花名册与仓库绑定；配置时区/权重；查看热力图与明细；多选开发者简单对比 |

## 4. 核心概念

### 4.1 Developer（开发者）与身份匹配

| 字段概念 | 说明 |
|:---------|:-----|
| **name** | 显示名（Display name），与 alias **分开存储** |
| **alias** | 短标识；与设置中的邮箱后缀拼成候选邮箱 |
| **tags** | 多对多 → 独立 Tag 实体 |
| **teams** | 多对多 → Team |

#### 匹配规则（已确认）

本机 `az` 实测（多仓 PR 作者/个人 reviewer）：**人类身份的 `uniqueName` 几乎全是邮箱**；非邮箱与 group 为 bot/服务账号/评审组，**不计开发者活跃度**，视为边缘情况不特殊建模。

| 规则 | 说明 |
|:-----|:-----|
| **主匹配** | 对每个配置的 `emailSuffix`，构造 `alias + "@" + suffix`，与事件中的 **`uniqueName` 忽略大小写精确相等** 则命中该 Developer |
| **后缀** | 设置页可配置**一个或多个**后缀（默认如 `microsoft.com`）；需要时再加其它域 |
| **未命中** | 不计到任何 Developer（或进入未映射列表供补 alias/后缀）；不猜测 |
| **忽略** | `isContainer` group、非邮箱 `uniqueName`（bot 等） |

不强制以 ADO identity id/descriptor 作为管理主键；实现上可选在命中后缓存 `uniqueName` 便于展示，但不替代 alias+后缀流程。

### 4.2 Repo（仓库）

- 绑定粒度：**单个 Git repository**。
- 一期 **provider = Azure DevOps**；GitHub **二期**。
- 绑定存 **D1**；采集在本机对已绑定列表执行。
- 必须支持**多 organization / 多 project**（单 org 假设不成立）。文档与示例使用**通用占位**，**不**写入真实内部 org 名。

### 4.3 Team（团队）

- 开发者与 Team：**多对多**。
- 用于组织与筛选；不与 Tag 混用。

### 4.4 Tag（标签）

- **独立实体**，含名称与**颜色**。
- 赋予 Developer，用于筛选与对比勾选。

### 4.5 Activity（活动）

- 某开发者在可归属时间点上的一条可计分事件（可带 repo / work item 引用）。
- 一期来源：ADO **PR** 与 **Work Item**（含个人 vote，见 §6）。
- **稳定性高于完整率**：无法用文档规定的稳定数据源重建的事件，**直接丢弃**，禁止猜测。

### 4.6 Score（得分）

- 在设置时区下，**开发者 × 自然日** 的积分（由当日 Activity 加权；物化与否见后续 schema 文档）。
- 权重在设置页可调；§6.3 为默认值。

### 4.7 时间

- 自然日边界由设置中的**时区**决定；**默认 `Asia/Shanghai`**。

### 4.8 设置（Settings）

| 项 | 说明 |
|:---|:-----|
| 邮箱后缀列表 | 与 alias 拼成候选邮箱，对齐 `uniqueName`（可多个后缀） |
| 时区 | 默认上海 |
| 活跃度权重 | 覆盖默认权重表 |

## 5. 实体层级（概念模型）

```
Instance（单次部署）
├── Settings                 # 含 emailSuffixes[]、时区、权重
├── Developer                # name + alias；靠 alias@suffix ≡ uniqueName 匹配
│     ├── tags[] → Tag
│     └── teams[] → Team
├── Team
├── Tag
├── Repo
├── Activity                 # 仅管线写入；Web 只读
└── Score                    # 仅管线写入；Web 只读
```

### 5.1 CRUD 与读写边界

| 数据 | Web | 本机管线 |
|:-----|:----|:---------|
| Developer / Team / Tag / Repo / Settings | **CRUD** | 可读（采集用绑定与 alias 映射） |
| Activity / Score | **只读** | **唯一写入方**（校验通过后 ingest） |

原则：先实体 CRUD，再关联；schema 中立，ADO/GitHub 经映射层进入。

### 5.2 预留扩展（空实现）

| 模块 | 意图 |
|:-----|:-----|
| **Livesite** | 线上事故相关贡献 |
| **Teams / IM** | IM 协作信号 |

扩展方式：新增 `Activity.type` + 可选权重键。

## 6. 一期活跃度（ADO only）

### 6.1 计入范围与**稳定数据源**（审查修订）

> 下列「稳定数据源」是硬约束。若实现走捷径（例如只用 PR 列表上的 `reviewers[].vote` 却无时间、或只用 WI `ChangedDate` 伪造逐日 updated），**视为不符合本定位**。

| 概念 type | 是否一期 | 稳定数据源（必须） | 说明 |
|:----------|:---------|:-------------------|:-----|
| `pr.merged` | ✅ | PR 的 `closedDate` + `status=completed` + 合并完成语义（如 merge commit / completion 字段） | 终态日计给作者 |
| `pr.closed` | ✅ | `closedDate` + 关闭且未合并 | 终态日计给作者 |
| `pr.created` | ✅（建议） | `creationDate` | 创建日；比不可还原的「历史 draft」更稳 |
| `pr.vote` | ✅ | **PR threads**（或官方带 **投票时间** 的 API）；**禁止**仅用当前 `reviewers[].vote` 快照反推历史日 | 记到**投票者**当日；忽略 group/container |
| `pr.active` | ⚠️ 可选/延后 | **PR iterations** 的 `updatedDate`（或同等带时间戳的 push/iteration 列表） | **禁止**用「当前 isDraft / 无时间戳的列表项」编造 |
| `pr.draft` | ❌ 一期不做「历史某日为 draft」 | `isDraft` **仅当前状态**，无法还原历史日 | 若需草稿信号：仅允许「**今天**仍为 draft 且当日有 iteration 更新」等可证伪规则；默认权重表可不收录 |
| `wi.created` | ✅ | WI `System.CreatedDate` | |
| `wi.closed` | ✅ | WI 关闭态对应的稳定时间字段（ClosedDate 或 revision） | |
| `wi.updated` | ✅ | **updates / revisions** 流（带修订时间）；**禁止**仅用当前 `ChangedDate` 当作「每一天都更新过」 | 按修订日生成；可对「同一 WI 同一自然日」去重 |

一期**不包含**：commits、GitHub、IM、Livesite 业务。

### 6.2 稳定性原则

1. **可重建**：任意 Activity 必须能从原始 JSON（落盘 raw）用确定性算法复算。  
2. **缺时间戳 = 不计分**：vote 无时间、active 无 iteration 时间 → **丢弃**，不写 Activity。  
3. **少而准**：完整率服从正确性。  
4. **个人 vote**：只计非 container 的个人 reviewer；group 票忽略。

### 6.3 默认权重

| Activity type | 默认权重 | 备注 |
|:--------------|--------:|:-----|
| `pr.merged` | 10 | |
| `pr.closed` | 2 | 关闭未合并 |
| `pr.created` | 2 | 创建 |
| `pr.vote` | 3 | 需 threads（或带时间 API） |
| `pr.active` | 2 | **仅当**有 iteration `updatedDate`；否则类型不出现 |
| `wi.created` | 3 | |
| `wi.updated` | 1 | 来自 revisions；同 WI 同日最多一次（默认） |
| `wi.closed` | 5 | |

**同日同一 PR 作者侧优先级（默认）**：`merged` > `closed` > `created` / `active`（终态优先；`pr.vote` 记投票者，独立）。

### 6.4 展示与对比（一期）

| 能力 | 要求 |
|:-----|:-----|
| 单人详情 | 热力图 + 明细 + Recharts 基础图 |
| **多选对比** | **必须**；一期简单比较即可 |
| Tag / Team 筛选 | 缩小多选集合 |

## 7. 数据管线：CLI / Scripts / Skills

### 7.1 分工

| 层 | 做什么 |
|:---|:-------|
| **CLI** | 语义命令：检查 `az` 登录、拉 PR/threads/iterations、拉 WI revisions、归一化输出 |
| **Scripts** | 串流程、目录约定、**schema 校验**、ingest D1 |
| **Skills** | 只编排命令，不定义 schema |

`gitinfo` / `pulse`：工程标杆（DI、JSON-first、高覆盖、清晰 exit code）。ADO 主路径可新建语义 CLI。

### 7.2 落盘与校验（硬约束）

1. 仅 JSON；带 `schemaVersion`。  
2. 目录示例（名以管线文档为准）：

   ```text
   .data/                          # gitignore，禁止提交
     raw/ado/{org}/{project}/
       repos/{repo}/
         prs/…
         pr-threads/…
         pr-iterations/…
       workitems/…                 # WI 属 Project，不属 Repo
       workitem-updates/…
     normalized/activities/…
     meta/runs/…
   ```

   - **PR 相关**：按 `{org}/{project}/repos/{repo}/…` 落盘（仓库作用域）。  
   - **Work Item**：按 `{org}/{project}/workitems/…` 落盘（**项目作用域**）；多个绑定仓库同属一个 Project 时 **只采集一份**，用 project 级外部键去重，禁止按 repo 重复拉 WI。

3. 落盘后 **必须** 验证；失败禁止 ingest。  
4. **`.data/` 必须在 `.gitignore` 中**（仓库已忽略）。

### 7.3 ADO 认证

- 本机 `az` + azure-devops 扩展；入口先校验登录。  
- 一期不做云端存 PAT 拉数。

## 9. 工程质量

| 项 | 标准 |
|:---|:-----|
| 方法论 | TDD |
| Lint | 0 error / 0 warning |
| 覆盖率 | **语句 / 行 / 函数 ≥ 95%**（vitest `thresholds` **硬门禁**，`test:coverage` 不达标即失败） |
| 分支覆盖 | **Web（非 View）≥ 95%**；既有 CLI 包以当前实测为 **floor 门禁防回退**（gitinfo branches ≥ 88、pulse ≥ 90），新增模块目标 ≥ 95% |
| 范围 | CLI、Scripts、Web 的 **Model + ViewModel / lib** 强制；**View 排除**（如 `App.tsx`、`**/views/**`） |
| Schema | 落盘 JSON 自动校验，防 Skill 漂移 |

门禁必须写在各包 `vitest.config` 的 `coverage.thresholds` 中，禁止仅文档声明。

## 10. 一期范围与非目标

### 10.1 一期做

- 实体 CRUD；alias + 后缀列表匹配 `uniqueName`  
- 本机 ADO 采集（稳定源见 §6.1）→ Activity → Score  

- JSON 落盘 + 校验 + ingest D1  
- Web：管理面 + 热力图/明细/简单多选对比 + 设置  
- Livesite / Teams：预留空实现  

### 10.2 一期明确不做

- 多租户 SaaS  
- GitHub 采集（二期）  
- 云端 PAT 拉 ADO  
- 用无时间戳 vote 快照 / 仅 ChangedDate / 仅 isDraft 伪造历史 Activity  
- Commit 级活跃度  
- 完整 Livesite / IM  
- 旧 AI Review 叙事  
- 移动端  
- 实时 Webhook 计分（一期批量时间窗）
